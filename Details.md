# Архитектура и технические детали: MAX Greeting Bot

## Схема взаимодействия компонентов

> **webhook** — MAX Messenger → FastAPI (Render) на Python 3.11 → HTTPS reply

| Компонент | Назначение |
|-----------|------------|
| GigaChat | генерация текста |
| GigaChat / Kandinsky | генерация картинки |
| SQLite | состояние, история |
| SMTP | отправка email |
| /image/ | хостинг картинок |

---

## Поток одного сообщения (что происходит за кадром)

1. **Пользователь жмёт кнопку в MAX** → MAX шлёт HTTPS POST на наш `/webhook`.

2. **FastAPI отвечает `200 OK` мгновенно** и ставит обработку в фоновую задачу (чтобы MAX не слал повторы).

3. **Фоновая задача:**
   - читает «состояние» пользователя из SQLite (какой шаг, какой повод, какой стиль, что пользователь ввёл),
   - вызывает GigaChat для генерации текста (OAuth2 токен кешируется на 30 минут),
   - вызывает GigaChat с `function_call=auto` для картинки — тот возвращает `<img src="file_id">`, мы качаем её по `/files/{id}/content`,
   - сохраняет картинку в нашу БД и выдаёт ссылку `/image/{id}.jpg`,
   - шлёт пользователю сообщения через MAX API с кнопками.

### 1. «Полный сценарий»

> `/start` → выбор повода (кнопки с динамическим списком под сегодняшнюю дату + «Свой повод» + «История») → инфо о получателе (или «Пропустить») → выбор стиля (5 вариантов) → генерация (текст от GigaChat-2-Max быстро, картинка отдельным сообщением до минуты) → превью с кнопками «Перегенерировать текст / Новая картинка / Сменить стиль / Сменить повод / Подтвердить» → выбор канала (MAX / email / ⏰ запланировать) → ввод контакта → отправка → кнопки «Ещё кому-то / Новое / История / Завершить».

### 2. «Как именно интегрирован GigaChat?»

- **Два раздельных вызова.** Один для текста, второй для картинки — чтобы текст не ждал долгую картинку.
- **Аутентификация:** OAuth2 на `ngw.devices.sberbank.ru:9443/api/v2/oauth` с Basic-ключом, scope `GIGACHAT_API_B2B`. Токен кэшируется, перевыпускается за минуту до истечения.
- **Модель:** `GigaChat-2-Max` (другие — GigaChat, GigaChat-2 — возвращают `402 Payment Required` на нашем ключе).
- **Картинки** через `function_call=auto` — модель сама инициирует вызов генератора изображения, ответ содержит `<img src="<file_id>">`, качаем байты по `/api/v1/files/{id}/content`.
- **Ретраи:** 2 попытки при таймауте (120 сек на каждую). При падении — понятное сообщение пользователю.

### 3. «Покажите промпт-инжиниринг»

**Тексты:** `TEXT_SYSTEM` задаёт персону («копирайтер Мэрии Казани, без штампов, 2-3 эмодзи максимум»). В промпт добавляются:
- Повод + стиль
- Ценности Мэрии Казани (слоган, 5 принципов) — только для стилей `corporate`/`official`
- Сегодняшняя дата → модель в P.S. упоминает другое событие дня
- Контекст о получателе (хобби, роль) — GigaChat вплетает естественно
- Инструкция формата: 3–5 предложений, до 550 символов

**Картинки:** структурированный промпт с явным поводом **КАПСОМ** («Повод — ПОВЫШЕНИЕ»), один визуальный хинт из описания получателя (любит кофе → чашка кофе, маркетолог → графики роста), один стилистический хинт, negative-prompts типа *«НЕ рисуй торт, свечи, воздушные шары»* для не-рождённых поводов — это убирает типичные галлюцинации модели в сторону стандартной открытки с тортом.

### 4. «Как обрабатываете ошибки?»

- **GigaChat упал на тексте** → «⚠️ GigaChat не смог сгенерировать текст: <причина>. Попробуй ещё раз».
- **Картинка не уложилась за 120 сек × 2 попытки** → «⚠️ Открытку не удалось. Текст готов — можно подтвердить без неё».
- **Неверный email** → регулярка + ответ «❌ Это не email, введи name@domain.ru».
- **Нестабильный MAX** (upload endpoint был ненадёжен) → хостим картинки у себя на `/image/{id}.jpg`, передаём URL.
- **Стыковка канала не удалась** (SMTP отвалился) → текст ошибки пользователю без спама.
- **Клик по старой кнопке** (из прошлого диалога) → «⚠️ Эта кнопка из старого сообщения, напиши /start» — защита от неконсистентного состояния.

### 5. «Мультиканальность?»

- **MAX** — через POST `/messages` с `chat_id` пользователя-получателя, картинка по URL нашего хостинга.
- **Email** — SMTP (дефолт Яндекс SSL :465) с HTML-письмом и inline-картинкой через CID.
- **Отложенная отправка** — модель `ScheduledGreeting`, парсер дат формата `30.04`, `30.04 14:30`. Очередь обрабатывается на каждом webhook-запросе + есть `/admin/tick` для cron.

### 6. «Архитектура и масштабируемость»

- **Один бинарь, один сервис** — проще деплоить.
- **Абстрактные слои:** `gigachat.py` знает только про GigaChat, `max_client.py` — только про MAX. Чтобы добавить Telegram — пишем новый клиент с тем же API, `flow.py` менять не надо.
- **Разграничение:** `prompts.py` отвечает за текстовый дизайн — редактор может править без знания Python (есть файл `CONTEXT.md` для ввода в курс).
- **Добавить новый повод** — 1 строка в `OCCASIONS_CONFIG` + описания в `IMAGE_DESCRIPTIONS`.
- **Добавить новый стиль** — 1 строка в `STYLES` + визуальные хинты.
- **Состояние в SQLite** с эфемерным хранилищем + ручные миграции через `ALTER TABLE` — для прода вынести в Postgres.

### 7. «Безопасность данных»

> Все токены, пароли, ключи — только через env-переменные (Render Environment), ни одной строки в коде.

- **HTTPS** на Render в комплекте.
- Пользовательские данные — имя/информация о получателе — хранятся в своей SQLite, не уходят наружу кроме GigaChat и MAX (обе **РФ-инфраструктуры**, что критично для муниципалитета).
- Картинки кешируются локально (в нашей БД), а не на CDN-третьих.
- *Что бы ещё усилил в проде:* whitelist `user_id` сотрудников Мэрии + проверка подписи webhook’а MAX.

### 8. «Что НЕ сделано / куда развивать?»

- **SQLite эфемерный** (Render free tier без диска) — на прод нужен Postgres или Persistent Disk.
- **Бесплатный Render засыпает** → запланированные отправки могут запоздать; привязать `cron-job.org` к `/admin/tick` или перейти на платный тариф без сна.
- **Справочник сотрудников Мэрии** (выбор получателя из списка, а не ввод контакта руками).
- **История** — только у каждого юзера своя; можно добавить общую аналитику для HR.
- **A/B-тестирование промптов** — сейчас вариативность случайная, можно собирать feedback и учить «что заходит».
